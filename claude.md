# Claude Project Instructions (claude.md)

このドキュメントは、このリポジトリにおける開発・運用・企画のコンテキストとルールをまとめたものです。コード生成、レビュー、ドキュメント作成の際はこのルールに従ってください。

## 1. ビジネス・企画の前提知識
* **プロジェクト概要**: 20〜30代のフリーランス向けタスク管理SaaSアプリ。
* **トーン＆マナー**: フレンドリーかつプロフェッショナル。専門用語を多用せず、直感的で分かりやすい言葉遣いを使用すること。
* **ブランドガイドライン**:
  * プライマリーカラー: `#4A90E2` (Blue)
  * アクセントカラー: `#F39C12` (Orange)
  * **基本文字色**: `#333333`
  * **背景色**: ダークモード時は `#1E1E1E`、ライトモード時は `#F5F5F5` を使用すること。
  * **フォント**: 常に `'Noto Sans JP', sans-serif` を指定し、一般的なシステムフォントにフォールバックさせないこと。

## 2. アーキテクチャと設計の決定事項
* **モジュールシステム**: CommonJSは使用せず、すべて **ES Modules (ESM)** を使用すること。
* **状態管理**: グローバルな状態管理にはReduxではなく、Zustandを使用している。
* **環境変数とセキュリティルール**:
  * **シークレットのハードコード禁止**: APIキー、トークン、パスワードなどの機密情報をコード内に直接書き込まないこと。必ず環境変数（`process.env.*`）経由で取得すること。
  * **フロントエンドへの露出防止**: バックエンド専用の機密情報には、フロントエンド公開用のプレフィックス（例: `NEXT_PUBLIC_` や `VITE_` など）を絶対に付けないこと。
  * **.env.exampleの更新**: 新しい環境変数を導入する提案をする際は、合わせて `.env.example` （または `.env.template`）ファイルにダミー値を記載する変更も提案に含めること。

## 3. バックエンド・データベースの規約
* **データベースのトランザクション管理ルール**:
  * 複数のテーブルをまたいで更新・作成を行う処理は、必ずデータベースのトランザクション機能でラップし、データの不整合を防ぐこと。
  * トランザクション処理は可能な限り短く保ち、外部API通信などの重い処理をトランザクション内に含めないこと。
* **エラーハンドリングの方針**:
  * データベースの生のエラーメッセージ（SQL構文エラーや内部構造など）をそのままクライアントに返却しないこと。
  * 必ずプロジェクト指定のカスタムエラークラスを使用し、標準化されたフォーマット（例: `{ error: { code, message } }`）に変換して返すこと。

## 4. コーディング規約・リポジトリのルール
* **ブランチ命名規則**: `{feature|bugfix|hotfix}/チケット番号-短い説明` (例: `feature/TASK-123-add-login-button`)
* **PR作成のルール**:
  * コミットメッセージは `Conventional Commits` に従うこと。
  * PRのディスクリプションには、必ず変更の目的とテスト手順を記載すること。
* **スタイリング**: Tailwind CSSを使用。カスタムCSSは極力書かず、ユーティリティクラスで完結させること。

## 5. 禁止事項と注意点（Must Not）
* **機密情報のログ出力禁止**: デバッグ目的であっても、APIキーやパスワードを含む環境変数の値を `console.log` 等で出力するコードを書かないこと。
* **レガシーコードの保護**: `src/legacy/` 配下のファイルは既存の動作を維持するため、明示的な指示がない限りリファクタリングや修正を行わないこと。
* **不要なコンソール出力**: 本番ビルドに含まれるコードに `console.log` や `console.debug` を残さないこと。必要な場合はカスタムロガーを使用すること。
* **モックの作成**: バックエンドAPIとの通信部分において、一時的なモック実装（ハードコードされたJSONなど）は勝手に作成しないこと。常に実際のAPIスキーマに基づくこと。
* **破壊的変更**: データベーススキーマを変更する提案をする際は、必ずマイグレーションへの影響や既存データへの対応を先に警告すること。

## 6. 開発・テストの主要コマンド
パッケージマネージャーには **pnpm** を使用しています。npm/yarnは使用しないでください。

* **依存関係のインストール**: `pnpm install`
* **ローカルサーバー起動**: `pnpm run dev`
* **ビルドの実行**: `pnpm run build`
* **Lint・フォーマット**: `pnpm run lint` & `pnpm run format`
* **テストの実行**: `pnpm run test` (ユニットテスト) / `pnpm run test:e2e` (E2Eテスト)
